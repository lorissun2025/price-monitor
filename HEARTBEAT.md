# HEARTBEAT.md
# 心跳检查清单 - 每次心跳都要执行

## ⚠️ 紧急：工作监控

### 🔍 检查当前任务状态（每次心跳必做）

**1. 有活跃任务吗？**
   - 有 → 进入【项目监控】
   - 无 → 进入【待机检查】

**2. 上次汇报是什么时候？**
   - < 4 小时 → 继续
   - > 4 小时 → 立即汇报当前进展
   - > 8 小时 → 异常！必须立即联系老板

**3. 有正在运行的子代理吗？**
   - 有 → 检查状态
   - 无 → 继续

---

## 📊 项目监控（有活跃任务时）

### 检查清单：
- [ ] 任务是否还在进行？
- [ ] 有没有卡住/失败？
- [ ] 是否超过预计时间？
- [ ] 需要老板决策吗？

### 决策逻辑：
```
任务超时 > 30% → 报告延期风险
任务超时 > 50% → 停止工作，询问老板
遇到阻塞问题 → 立即说明 + 暂时计划
遇到理解错误 → 立即澄清
```

### 汇报内容模板（超时汇报）：
```
📊 [项目名称] 进展汇报

✅ 已完成：[已完成的部分]
🔄 当前进度：[百分比]
⏰ 预计完成：[时间]
🚨 情况：[正常/延期/遇到问题]

下一步：[下一阶段计划]
```

## ✅ 心跳完成检查

每次心跳结束时，确保：
- [ ] 知道当前在做什么
- [ ] 上次汇报时间 < 8 小时
- [ ] 有任务时知道进度
- [ ] 无任务时知道下一步
- [ ] 更新了 heartbeat-state.json

---

**核心原则：**
1. 永远不要让老板超过 8 小时不知道我在做什么
2. 有任务必须主动汇报，不能等老板问
3. 遇到问题第一时间说明，不要隐瞒
4. 无任务时主动建议，不要傻等

**记住：沉默 ≠ 努力工作，沉默 = 老板不知道我在做什么！**

## 🔌 网络连接健康检查（新增）

### ⚠️ WS 断线检测（2026-03-18 事件复盘）

**问题现象：**
- 主 WebSocket 连接持续断线（TLS 握手失败）
- Session 实际在执行任务（exec + write 成功）
- 但消息无法推送回来 → 用户看到"没反应"
- Gateway 多次重启尝试恢复，但 WS 仍挂

**检测指标：**
```
如果发现以下情况 → 可能是 WS 断线：
1. 最近心跳响应时间 > 30 秒
2. 自己的消息发送失败（network/socket 错误）
3. 收不到任何系统消息（正常情况下应该有）
4. 已知有任务在进行，但看不到任何进展更新
```

**本地日志记录（WS 断线时）：**
即使无法通知用户，我应该记录到本地文件：
```
memory/ws-disconnect-events.md
---
## [日期时间] WS 断线检测

**症状：**
- [具体症状]

**本地状态：**
- 任务名称：[如果有]
- 任务进度：[已知进度]
- 已完成操作：[exec/write 成功的操作]

**行动：**
1. 记录本地日志
2. 持续尝试重连
3. 任务继续执行（如果 gateway 正常）
4. 恢复后第一时间补发进展汇报
```

**恢复后的补救措施：**
```
✅ WS 恢复连接

【断线期间完成的工作】
1. [任务1]
2. [任务2]

【断线时间】从 [开始时间] 到 [结束时间]

【当前状态】[简要说明】

抱歉让您看到"没反应"，其实我一直在工作，只是消息推不回来！🙏
```

**根本原因分析（2026-03-18 事件）：**
```
可能原因：
1. 公司网络代理/防火墙拦截 TLS 流量
2. VPN/代理切换导致连接中断
3. OpenClaw 中心服务器网络抖动
4. 证书验证失败

建议排查：
1. 检查是否开启了 VPN/代理
2. 检查公司网络是否有防火墙规则变更
3. 检查 Gateway 日志：openclaw gateway status
4. 尝试重启：openclaw gateway restart
```
