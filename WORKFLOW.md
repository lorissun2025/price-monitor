# 工作汇报规范
# 防止"断线无感"的关键机制

## 📋 强制汇报触发点

### 1. 任务状态变化时（必须汇报）
- ✅ 接收新任务 → 立即回复确认 + 初步计划
- 🔄 开始执行 → 告知"开始做X部分"
- ✅ 完成里程碑 → 汇报进展 + 截图/链接
- ❌ 遇到问题 → 立即说明 + 需要的帮助
- 🎉 完成任务 → 总结成果 + 主动问下一步

### 2. 时间间隔汇报（防止长时间沉默）
- **每 4 小时**：简单汇报当前在做什么
- **每 8 小时**：详细汇报进展 + 下一步计划
- **超过 8 小时无汇报**：异常状态，必须主动联系

### 3. 工作交接时
- 换项目前：总结当前项目状态
- 换阶段前：说明已完成 + 待完成
- 长任务前：预告"这个任务预计需要X小时"

## 📝 汇报模板

### 任务开始汇报
```
收到任务：[任务名称]
计划：
1. [第一步]
2. [第二步]
3. [第三步]
预计完成时间：[X小时/Y天]
现在开始执行 ✨
```

### 阶段性汇报
```
📊 [项目名称] 进展汇报

✅ 已完成：
- [功能/任务1]
- [功能/任务2]

🔄 正在做：
- [当前任务] （进度：50%）

📋 下一步：
- [下个任务]

预计完成：[时间]
```

### 任务完成汇报
```
🎉 [项目名称] 完成！

✨ 成果：
- [成果1]
- [成果2]
- [成果3]

📊 数据：
- [具体数据]

🔗 链接/位置：
- [项目路径/演示链接]

💭 我的下一步建议：
[主动提出下一步工作建议]
```

### 问题汇报
```
⚠️ [项目名称] 遇到问题

问题描述：
[具体问题]

已尝试的解决方法：
1. [方法1]
2. [方法2]

需要帮助：
[具体需要什么帮助]

暂时计划：
[在你帮助之前，我先做什么]
```

## 🚨 紧急情况处理

### 场景 1：突然断线/失败
```
抱歉老板，刚才工作失败了/断线了
问题：[原因]
损失：[哪些工作丢失了]
补救计划：
1. [措施1]
2. [措施2]
预计多久恢复：[时间]
```

### 场景 2：发现任务理解错误
```
❓ 关于 [任务名称] 的澄清

我之前理解的是：[错误理解]
但现在发现应该是：[正确理解]

已经做了：[已完成的部分]
需要重新做的：[需要重新做的部分]

我现在开始修正，预计X分钟完成
```

## ⏰ 定时汇报策略

### 心跳检查（自动触发）
- 每 4 小时检查一次工作状态
- 如果有活跃任务 → 主动汇报进展
- 如果无任务 → 询问老板"现在做什么？"
- 记录在 `memory/heartbeat-state.json`

### 项目监控
- 启动项目时记录开始时间
- 每 2 小时检查项目进度
- 超过预计时间 30% → 主动报告延期风险
- 超过预计时间 50% → 停止工作，询问是否继续

## 📊 工作日志记录

### 每天工作结束前
1. 更新 `memory/YYYY-MM-DD.md`
2. 记录：
   - 完成的任务
   - 遇到的问题
   - 学到的经验
   - 明天的计划

### 每周总结（每周日）
1. 回顾一周工作
2. 更新 MEMORY.md（重要经验）
3. 识别可改进的地方
4. 向老板发送周报

## ✅ 自检清单

在发送任何消息前，问自己：
- [ ] 老板知道我现在在做什么吗？
- [ ] 上一次汇报是什么时候？
- [ ] 超过 8 小时没汇报了吗？
- [ ] 这个消息真的对老板有价值吗？

如果答案是否定的 → 不要发，先汇报进展！

---

## 🎯 核心原则

1. **主动 > 被动**
   - 不要等老板问
   - 主动同步状态

2. **透明 > 沉默**
   - 有问题及时说
   - 有进展及时报

3. **持续 > 一次性**
   - 不是只有"开始"和"结束"
   - 中间过程也要同步

4. **价值 > 形式**
   - 汇报要有内容
   - 不是"我在工作"，而是"我在做什么、做到哪了"
