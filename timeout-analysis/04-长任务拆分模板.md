# 长任务拆分模板
## 避免单 run 超时 - 可复用任务拆分方案

**模板版本**: v1.0
**创建时间**: 2026-03-17
**适用场景**: 任何可能超过 150 秒的任务

---

## 📋 模板概览

### 核心理念

**大任务 = 摘要 → 分步执行 → 汇总**

将一个复杂、耗时的任务拆解为多个小任务，每个小任务独立执行，最终汇总结果。

### 为什么需要拆分？

| 问题 | 不拆分 | 拆分后 |
|------|--------|--------|
| 超时风险 | 单次超时 = 整体失败 | 单次超时 = 只影响一个子任务 |
| 等待时间 | 可能等 10 分钟 | 每步 < 2 分钟 |
| 失败率 | 100% 失败 | 部分失败，可重试 |
| 进度可见 | 只有"完成/失败" | 每步都有进度 |
| 资源浪费 | 超时后全丢 | 中间结果已保存 |

### 适用场景

✅ **需要拆分**：
- 阅读并分析 5+ 个文件
- 批量处理 10+ 个文件
- 复杂代码重构（> 1000 行）
- 多阶段系统设计
- 生成大型技术文档
- 数据分析 + 可视化
- API 调用 + 处理 + 存储

❌ **不需要拆分**：
- 简单问答（< 10 秒）
- 单文件操作（< 30 秒）
- 单次 API 调用
- 快速查询（< 30 秒）

## 🔄 标准拆分流程

### 三阶段模板

#### 阶段 1: 摘要阶段（30-60秒）

**目标**: 理解任务，识别子任务，制定计划

**执行步骤**:
```
1. 读取任务输入（文件、需求文档、URL 等）
2. 快速扫描关键信息
3. 识别主要组件/步骤
4. 生成任务摘要（< 500 字）
5. 列出子任务清单（3-10 个）
6. 保存到临时文件
```

**输出示例**:
```json
{
  "taskSummary": "重构 hotspot-map 项目，从原生 JS 迁移到 React",
  "subtasks": [
    {
      "id": "1",
      "name": "初始化 React 项目",
      "estimatedTime": "30秒",
      "dependencies": [],
      "output": "react-project/"
    },
    {
      "id": "2",
      "name": "迁移地图组件",
      "estimatedTime": "60秒",
      "dependencies": ["1"],
      "output": "src/components/Map/"
    },
    {
      "id": "3",
      "name": "迁移筛选组件",
      "estimatedTime": "45秒",
      "dependencies": ["1"],
      "output": "src/components/Filter/"
    }
  ],
  "totalEstimatedTime": "135秒",
  "checkpoint": "task-summary.json"
}
```

**Agent 提示词**:
```
你正在执行任务的【摘要阶段】。

任务：[用户输入的任务描述]

请执行以下步骤：
1. 快速阅读相关文件/文档
2. 理解任务目标
3. 识别需要执行的子任务（3-10个）
4. 估算每个子任务的时间
5. 识别子任务之间的依赖关系

输出格式：JSON（如上示例）

时间限制：60秒
超时处理：如果超时，直接输出当前已识别的子任务清单
```

---

#### 阶段 2: 执行阶段（每个子任务 60-120秒）

**目标**: 逐个执行子任务，每步独立可重试

**执行策略**:
```
并行 vs 串行：
- 无依赖：可并行执行（最多 3 个同时）
- 有依赖：串行执行，按顺序完成

每个子任务：
1. 读取子任务定义
2. 执行具体操作（读/写/编辑/调用 API）
3. 保存结果到临时文件
4. 标记完成状态
5. 遇到错误：记录但继续下一个
```

**输出示例**:
```json
{
  "subtaskId": "2",
  "name": "迁移地图组件",
  "status": "completed",
  "startTime": 1710653880,
  "endTime": 1710653940,
  "duration": 60,
  "output": {
    "filesCreated": ["src/components/Map/MapView.jsx", "src/components/Map/MapMarker.jsx"],
    "filesModified": [],
    "error": null
  },
  "checkpoint": "subtask-2-result.json"
}
```

**Agent 提示词**:
```
你正在执行任务的【执行阶段 - 子任务 {id}】。

子任务定义：
[JSON 格式的子任务定义]

请执行以下步骤：
1. 检查依赖是否满足
2. 执行具体操作
3. 保存结果到临时文件
4. 记录耗时和状态

时间限制：120秒
超时处理：
- 如果超时，保存当前进度
- 记录 "timeout" 状态
- 继续下一个子任务

错误处理：
- 如果遇到错误，保存错误信息
- 标记为 "failed" 状态
- 继续下一个子任务（除非是关键依赖）
```

**子任务执行器（伪代码）**:
```javascript
async function executeSubtask(subtask) {
  const startTime = Date.now();

  try {
    // 检查依赖
    for (const dep of subtask.dependencies) {
      if (!checkDependencyCompleted(dep)) {
        return { status: 'waiting', reason: 'dependency_not_satisfied' };
      }
    }

    // 执行具体操作
    const result = await performSubtask(subtask);

    // 保存结果
    await saveResult({
      subtaskId: subtask.id,
      status: 'completed',
      duration: Date.now() - startTime,
      output: result
    });

    return { status: 'completed', result };

  } catch (error) {
    // 保存错误
    await saveResult({
      subtaskId: subtask.id,
      status: 'failed',
      duration: Date.now() - startTime,
      error: error.message
    });

    return { status: 'failed', error };
  }
}
```

---

#### 阶段 3: 汇总阶段（60-120秒）

**目标**: 收集所有子任务结果，生成最终输出

**执行步骤**:
```
1. 读取所有子任务结果文件
2. 分析哪些完成/失败/超时
3. 合并成功的输出
4. 处理失败的部分（记录或重试）
5. 生成最终报告/文档/代码
6. 保存最终结果
7. 清理临时文件
```

**输出示例**:
```json
{
  "taskId": "task-1710653880",
  "status": "partial_success",
  "startTime": 1710653880,
  "endTime": 1710654300,
  "totalDuration": 420,
  "subtasks": {
    "total": 5,
    "completed": 4,
    "failed": 1,
    "timeout": 0
  },
  "finalOutput": {
    "path": "/path/to/final/result",
    "type": "file" | "code" | "report"
  },
  "failedSubtasks": [
    {
      "id": "3",
      "reason": "file_not_found",
      "retryRequired": true
    }
  ],
  "summary": "4/5 子任务成功完成，子任务 3 需要重试"
}
```

**Agent 提示词**:
```
你正在执行任务的【汇总阶段】。

子任务结果文件：
[列出所有子任务结果文件路径]

请执行以下步骤：
1. 读取所有子任务结果
2. 统计完成/失败/超时数量
3. 合并成功的输出
4. 分析失败原因，判断是否需要重试
5. 生成最终输出（代码/文档/报告等）
6. 保存最终结果

时间限制：120秒
超时处理：
- 如果超时，保存当前已汇总的结果
- 输出 "partial" 状态
- 记录哪些子任务尚未汇总

最终输出格式：
- 代码项目：生成完整的项目结构
- 文档：生成 Markdown 文档
- 报告：生成 JSON 或 Markdown 报告
```

## 📦 完整模板 - 复制即用

### 模板 1: 文件批量处理

**适用场景**: 批量读取、修改、分析多个文件

```markdown
# 任务：{任务名称}

## 阶段 1: 摘要（60秒）

1. 列出需要处理的所有文件路径
2. 对每个文件进行快速扫描（前 50 行）
3. 识别文件类型和内容结构
4. 生成处理计划

输出：`task-summary.json`

## 阶段 2: 执行（每个文件 60秒）

对每个文件：
1. 读取完整内容
2. 执行具体操作（修改/分析/转换）
3. 保存结果
4. 记录状态（success/failed/timeout）

并行执行：最多 3 个文件同时处理

输出：`subtask-{id}-result.json`（每个文件一个）

## 阶段 3: 汇总（90秒）

1. 收集所有文件处理结果
2. 生成汇总报告
3. 创建修改后的文件（如需要）
4. 清理临时文件

输出：`final-result.md` + `processed-files/`
```

### 模板 2: 代码重构

**适用场景**: 大型代码库重构（> 5 个文件）

```markdown
# 任务：{任务名称} - 代码重构

## 阶段 1: 分析（90秒）

1. 读取项目结构和主要文件
2. 识别需要重构的文件
3. 识别代码依赖关系
4. 制定重构计划

输出：`refactor-plan.json`

## 阶段 2: 执行（每个文件 120秒）

按依赖顺序逐个文件：
1. 读取文件内容
2. 应用重构规则
3. 验证语法正确性
4. 保存修改后的文件

串行执行（确保依赖正确）

输出：`refactored-{filename}.json`

## 阶段 3: 测试和汇总（120秒）

1. 运行测试（如果有）
2. 验证所有修改
3. 生成重构报告
4. 对比修改前后差异

输出：`refactor-report.md` + `diff.patch`
```

## 🔧 实现细节

### 临时文件命名规范

```
task-{taskId}-summary.json          # 阶段1 摘要
subtask-{taskId}-{subtaskId}.json   # 阶段2 每个子任务结果
task-{taskId}-final.json            # 阶段3 最终汇总
task-{taskId}-report.md             # 最终报告
checkpoint-{timestamp}.json         # 断点恢复（可选）
```

### 错误处理策略

| 错误类型 | 处理方式 | 是否继续 |
|---------|---------|---------|
| 文件不存在 | 记录，跳过 | ✅ 是 |
| 权限错误 | 记录，停止 | ❌ 否 |
| 语法错误 | 记录，保存当前状态 | ✅ 是 |
| 超时 | 保存部分结果，继续 | ✅ 是 |
| API 失败 | 重试 2 次，仍失败则跳过 | ✅ 是 |

### 进度跟踪

```json
{
  "progress": {
    "total": 100,
    "current": 45,
    "percentage": 45,
    "stages": {
      "summary": "completed",
      "execution": "in_progress",
      "finalization": "pending"
    },
    "subtasks": [
      { "id": "1", "status": "completed", "duration": 30 },
      { "id": "2", "status": "in_progress", "duration": null },
      { "id": "3", "status": "pending", "duration": null }
    ]
  }
}
```

### 断点恢复（可选高级功能）

如果任务被中断，可以从断点恢复：

```javascript
// 检查是否有断点文件
if (fs.existsSync('checkpoint-latest.json')) {
  const checkpoint = JSON.parse(fs.readFileSync('checkpoint-latest.json'));
  
  // 从中断的子任务继续
  resumeFromSubtask(checkpoint.lastCompletedSubtaskId);
}
```

## 🎯 使用建议

### 何时强制拆分？

- 任务描述包含"批量"、"多个"、"所有"等关键词
- 预估执行时间 > 2 分钟
- 涉及 5+ 个文件操作
- 包含 3+ 个独立步骤
- 容易超时/失败的任务类型

### 何时可以不拆分？

- 单文件、单操作任务
- 预估时间 < 1 分钟
- 强依赖、无法并行的任务
- 简单查询或计算

### 最佳实践

1. **小任务优先**: 子任务时长控制在 30-120 秒
2. **并行最大化**: 无依赖的子任务尽量并行
3. **结果持久化**: 每步都保存结果，避免丢失
4. **错误容忍**: 个别失败不影响整体
5. **进度透明**: 实时报告进度

## 🔄 与现有系统集成

### 在 OpenClaw 中使用

**自动拆分（推荐）**:

创建一个自动拆分工具 `task-splitter.js`:

```javascript
import { spawn } from 'child_process';

async function splitAndExecute(task) {
  // 阶段1: 摘要
  const summary = await executeStage('summary', task);
  
  // 阶段2: 执行（并行）
  const subtasks = summary.subtasks;
  const results = await executeSubtasks(subtasks);
  
  // 阶段3: 汇总
  const final = await executeStage('finalization', results);
  
  return final;
}

// 在 Agent 中调用
const task = {
  description: "批量处理 10 个文件",
  files: ["file1.js", "file2.js", ...]
};

const result = await splitAndExecute(task);
```

**手动使用**:

在提示词中明确要求拆分：
```
请使用"三阶段拆分模板"完成这个任务：
1. 先做摘要阶段（60秒）
2. 再做执行阶段（每个子任务 120秒）
3. 最后做汇总阶段（120秒）

任务描述：[具体任务]
```
