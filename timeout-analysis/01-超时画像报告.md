## 🔍 Top 10 高风险请求模式

基于日志分析，识别出以下高风险请求模式：

### 1. **长时间任务执行** (Risk: 🔴 极高)
- **模式**: 超过 10 分钟的复杂任务
- **典型案例**: EvoMap 任务执行、文件处理、代码生成
- **超时占比**: 约 80%
- **触发条件**: 单次 run 包含多个工具调用/文件操作

### 2. **大型文件操作** (Risk: 🔴 极高)
- **模式**: 大文件读取/写入/编辑
- **典型案例**: `read` 大文件、`write` 大型 JSON、`edit` 复杂代码
- **特征**: 文件大小 > 100KB 或代码行数 > 1000
- **超时原因**: I/O + AI 处理时间累积

### 3. **复杂逻辑推理** (Risk: 🟠 高)
- **模式**: 需要多轮思考的任务
- **典型案例**: 系统设计、算法分析、架构决策
- **特征**: reasoning 模式启用，上下文长
- **超时原因**: 模型推理时间长

### 4. **多工具链式调用** (Risk: 🟠 高)
- **模式**: 一次性调用 5+ 工具
- **典型案例**: 批量文件处理 + API 调用 + 数据分析
- **特征**: 单个工具调用超时但累计时间很长
- **超时原因**: 工具调用序列过长

### 5. **外部 API 请求** (Risk: 🟡 中)
- **模式**: 网络请求 + AI 处理
- **典型案例**: EvoMap API 调用、Feishu API、web_fetch
- **特征**: 网络延迟 + 模型处理
- **超时原因**: 外部服务不稳定或响应慢

### 6. **日志/大数据分析** (Risk: 🟡 中)
- **模式**: 读取和分析大量日志
- **典型案例**: grep 日志、统计错误、生成报告
- **特征**: 数据量大，需要多次工具调用
- **超时原因**: 数据读取 + 处理累积

### 7. **代码重构/大型编辑** (Risk: 🟡 中)
- **模式**: 使用 `edit` 修改大文件
- **典型案例**: 重构代码库、添加新功能
- **特征**: 文件大，edit 操作复杂
- **超时原因**: 精确匹配 + 上下文理解

### 8. **长上下文任务** (Risk: 🟡 中)
- **模式**: 输入包含大量上下文
- **典型案例**: 阅读多个文件、汇总信息
- **特征**: 上下文长度 > 50K tokens
- **超时原因**: 模型处理长文本慢

### 9. **周期性定时任务** (Risk: 🟠 高)
- **模式**: cron 触发的定期任务
- **典型案例**: EvoMap 自动接取、数据抓取
- **特征**: 固定时间触发
- **超时原因**: 任务复杂度未评估

### 10. **并发任务竞争** (Risk: 🟢 低)
- **模式**: 多个子代理同时运行
- **典型案例**: 8 个子代理同时工作
- **特征**: 资源竞争，响应变慢
- **超时原因**: maxConcurrent=8 导致资源紧张

## 📈 超时原因深度分析

### 根本原因

1. **超时时间过长 (600000ms)**
   - 10 分钟太长，用户等太久
   - 超时后直接 surface_error，没有降级/重试

2. **缺乏重试机制**
   - 超时后直接失败，100% surface_error
   - 没有指数退避/快速失败策略

3. **任务拆分不足**
   - 大任务单次 run 完成，没有分步执行
   - 缺乏"摘要 → 执行 → 汇总"流程

4. **没有降级模式**
   - 超时后没有简化版执行方案
   - 全部或全无，用户体验差

5. **缺少熔断机制**
   - 连续超时不触发熔断
   - 继续尝试，浪费资源

### 伴随噪音问题

#### 1. **tools.allowlist 警告**
```
[tools] tools.profile (coding) allowlist contains unknown entries (apply_patch, cron).
These entries are shipped core tools but unavailable in the current runtime/provider/model/config.
```
- **频率**: 每 10 分钟一次
- **影响**: 日志污染，不致命
- **原因**: 配置文件中的 allowlist 包含当前环境不可用的工具

#### 2. **Feishu Delivery 失败**
```
[delivery-recovery] Retry failed for delivery xxx: Cannot destructure property 'tenant_access_token' of '(intermediate value)' as it is undefined.
```
- **频率**: 多次重试失败
- **影响**: Feishu 消息无法送达
- **原因**: tenant_access_token 未正确获取/刷新

#### 3. **plugins.allow 警告**
```
[plugins] plugins.allow is empty; discovered non-bundled plugins may auto-load: feishu, channels.
```
- **频率**: Gateway 启动时
- **影响**: 安全风险（未知插件可能自动加载）
- **原因**: 未明确设置允许的插件列表

## 🎯 可执行建议

### 短期优化（立即执行）

#### 1. 调整超时时间
```json
// 在配置中添加
{
  "agents": {
    "defaults": {
      "timeoutMs": 150000  // 150秒 (2.5分钟)
    }
  }
}
```
**预期效果**:
- 减少用户等待时间（从 10 分钟降到 2.5 分钟）
- 快速失败，避免无效等待

#### 2. 启用重试机制
```json
{
  "agents": {
    "defaults": {
      "retry": {
        "maxAttempts": 2,
        "backoffMs": 1500,
        "retryOn": ["timeout", "5xx"]
      }
    }
  }
}
```
**预期效果**:
- 临时性超时可恢复
- 降低失败率到 < 10%

#### 3. 移除 allowlist 警告
```json
// 编辑 tools.profile 配置，移除 apply_patch, cron
{
  "tools": {
    "profile": "coding",
    "allowlist": []  // 清空或只保留确实可用的工具
  }
}
```
**预期效果**:
- 消除 allowlist 噪音日志

### 中期优化（1-2天内）

#### 4. 实现降级执行模式
- **检测**: 超时或接近超时时
- **降级策略**:
  - 减少上下文长度（摘要化）
  - 限制工具调用次数（最多 3 次）
  - 缩短回答长度（< 1000 字）
- **触发条件**: `elapsedTime > 0.7 * timeoutMs`

#### 5. 添加熔断机制
- **触发条件**: 5 分钟内连续 3 次超时
- **熔断时间**: 3-5 分钟
- **熔断期间**: 只允许轻量请求（< 30 秒）
- **自动恢复**: 3 分钟内无超时则恢复

#### 6. 实现任务拆分模板
创建可复用模板，自动拆分大任务：
```markdown
# 任务拆分模板

1. **摘要阶段** (30秒)
   - 读取输入
   - 生成任务摘要
   - 识别子任务

2. **执行阶段** (每子任务 < 2 分钟)
   - 逐个执行子任务
   - 每个超时独立重试
   - 实时保存中间结果

3. **汇总阶段** (60秒)
   - 收集所有结果
   - 生成最终输出
   - 保存到文件
```

### 长期优化（1周内）

#### 7. 优化 Feishu Delivery
- 修复 `tenant_access_token` 获取逻辑
- 添加 token 刷新机制
- 实现 fallback (存储到文件/数据库)

#### 8. 配置 plugins.allow
```json
{
  "plugins": {
    "allow": ["feishu", "channels"]  // 明确列出可信插件
  }
}
```

#### 9. 监控和告警
- 实时超时率监控
- 超时率 > 30% 时触发告警
- 自动降级到 flash 模型
